# 入口定位与多系统判定

## 推荐配合使用的场景

```
场景 1：在子系统目录中启动
  打开 nhai-product → 说"帮我整理简历" → skill 分析当前目录
  → 询问是否还有其他子系统 → 提供路径 → 聚合输出

场景 2：在无关目录中启动（最灵活）
  在任意位置（如 rule-main）→ 说"帮我提炼项目经历"
  → skill 询问路径 → 用户提供所有子系统路径 → 远程分析 → 聚合输出
  （无需切换工作区，无需打开项目目录）

场景 3：配合 JD 定制
  用户提供目标岗位 JD + 项目路径 → skill 分析匹配度 → 输出定制版简历

场景 4：单个子系统快速分析
  打开某个子系统 → 只分析这一个 → 快速输出
```

---

## 入口：工作区感知与目录定位

skill 启动后，**不假设用户一定打开了目标项目目录**。用户可能在任何位置触发此 skill。

### 步骤 1：判断启动上下文

检查当前工作区（Workspace Path），判断属于哪种情况：

**情况 A：当前工作区就是一个 Java 项目/子系统**
- 检测到 `pom.xml` / `build.gradle` / `src/main/java` 等 Java 项目标志
- 展示识别结果，询问用户：
```
检测到当前工作区是一个 Java 项目：
- 目录：D:/code/nhai-product
- 类型：微服务子系统（包含 nhai-product-api, nhai-product-server）

请选择：
1. 就分析当前这个子系统
2. 除了当前目录，还有其他子系统需要一起分析（请提供路径）
3. 不分析当前目录，我直接指定要分析的路径
```

**情况 B：当前工作区不是 Java 项目（如文档目录、规则仓库等）**
- 没有检测到 Java 项目标志
- 直接询问用户提供项目路径：
```
当前工作区不是 Java 项目。
请提供你想分析的项目路径（支持多个，每行一个）：

示例：
D:\11051426\IdeaProjects\nhai\bak-2\nhai-product
D:\11051426\IdeaProjects\nhai\bak-2\nhai-quotation
D:\11051426\IdeaProjects\nhai\nhai-infra\nhai-infra-starters\nhai-infra-boot-starter-lock
```

### 步骤 2：确定分析目标

无论从哪种情况进入，最终需要确定一个**目标目录列表**：

```python
# 伪逻辑
target_dirs = []  # 要分析的目录列表
if 用户选择分析当前目录:
    target_dirs.append(当前工作区路径)
if 用户提供了额外路径:
    target_dirs.extend(用户提供的路径列表)
```

### 步骤 3：确定分析模式

根据 target_dirs 的数量和关系：

**模式 A：单目录分析**
只有一个目标目录。
→ 直接对该目录执行第零步起的完整流程。

**模式 B：多子系统聚合（属于同一个大项目）**
多个目录属于同一个项目/仓库的不同子系统。
→ 分别扫描，最终合并为一份项目经历。

**模式 C：多个独立项目**
目录之间互不相关（不同仓库/不同公司）。
→ 各自独立输出一条项目经历，最后统一润色。

如何判断是 B 还是 C：
- 检查各目录是否在同一个 git 仓库下（`git rev-parse --show-toplevel`）
- 如果无法判断，直接问用户："这些目录是同一个项目的不同子系统，还是独立的项目？"

**关键设计原则**：
- **不要求用户打开任何特定目录**——从任何位置都能启动，只要提供路径
- 对于用户提供的路径，通过 Shell 工具的 `working_directory` 参数切换到对应目录执行命令
- 先验证路径是否存在且包含 Java 代码，不存在则提示用户检查
- 多子系统模式下，git 身份只需确认一次，复用到所有目录
- 每个子系统独立分析其技术栈和业务，不要混为一谈
- 如果某个目录分析后发现内容太薄（如只是配置项目、脚手架），提示用户是否跳过

## 第 0.5 步：多系统判定与拆分策略

用户可能在一个项目（仓库）中参与了多个子系统/微服务。需要判断并选择合适的简历呈现方式。

### 1. 识别子系统边界

**场景 A：用户打开的是某个子系统目录（最常见）**
- 当前目录本身就是一个子系统（如 `nhai-product`）
- 检查当前目录内部是否还有子模块（如 `nhai-product-api`、`nhai-product-server`）
- 子模块通常是同一服务的分层（API/Server/Common），不需要拆分为多条经历
- 询问："当前子系统 [名称] 中，你主要负责哪些模块？还是全部都参与了？"

**场景 B：用户打开的是项目根目录（较少见）**
- 扫描项目目录，检查是否为多模块结构（如 `xxx-order`、`xxx-user`、`xxx-gateway`）
- 检查是否有多个独立的 `pom.xml` / `build.gradle`（父子模块）
- 检查是否有多个独立的 `application.yml`（多个可独立部署的服务）
- 询问用户："你在这个项目中参与了哪几个子系统？"

### 2. 选择呈现方式（展示给用户选择）

**方式 A：一个大项目 + 按职责模块描述（推荐用于微服务项目）**
适用于：参与了同一平台的多个微服务，在简历上作为同一个项目展示
```
项目：xxx 平台（Spring Cloud 微服务架构，10+ 个服务）

核心贡献：
1. 【核心中台服务】设计了 xxx 引擎...
2. 【报价服务】负责 xxx 模块...
3. 【基础设施】设计了 xxx Starter...
```

**方式 B：按子系统拆分为多条项目经历**
适用于：参与了 2-3 个独立子系统，且各有可讲的亮点，适合简历篇幅较长的情况
```
项目一：xxx 平台 — 核心中台服务
职责：...

项目二：xxx 平台 — 报价引擎服务
职责：...
```

**方式 C：挑最亮的一两个写**
适用于：参与了多个系统但大部分是 CRUD，只有 1-2 个有技术深度
只挑有技术亮点的子系统深入写，其余在"工作职责"中一笔带过

### 3. 确认后进入扫描

- 用户选择呈现方式后，按所选方式组织后续的扫描和输出
- 如果需要扫描当前工作区之外的目录，通过 Shell 工具切换 working_directory 执行
