# 提炼与润色规则

## 第二步：提炼与结构化

将扫描结果整理为简历项目模板。**3 年经验的定位不是"初级开发"而是"能独立设计模块的中级开发"**，输出应体现这一层次。

### 输出模板

```
**项目名称**：xxx 平台（简短定性描述，如"微服务架构/SaaS平台/高并发系统"）
**项目简介**：一句话说明项目是什么 + 业务体量（用户量/数据量/服务数）（1-2行）
**技术架构**：JDK XX + Spring Boot x.x + [核心框架] + [数据层] + [中间件] + [部署方式]
**我的定位**：一个短语定性你的角色（如"核心模块 Owner""框架设计者""基础设施贡献者"）

**核心贡献**：（3-5 条，每条遵循 STAR-X 结构）

1. **[亮点标题]**（一个短语概括这是什么）
   - 背景：[业务挑战/技术痛点，1 句话]
   - 方案：[技术选型 + 关键设计决策 + 为什么这样选，2-3 句话]
   - 成果：[量化指标]
```

### 每条贡献的质量标准

- 有"为什么"（技术选型理由）而不只是"做了什么"
- 有量化结果（数字/百分比/对比值），无法量化的用 `[待补充: xx]` 标记
- 技术方案描述到"架构级别"而非"代码级别"（说设计思路，不要贴代码）
- 单条长度：3-5 行，不要写成论文

## 第三步：优化润色

### 3.1 STAR-X 法则

| 要素 | 含义 | 示范 |
|------|------|------|
| S - Situation | 项目背景/挑战 | "系统管理 30+ 种业务类型，状态流转规则各异" |
| T - Task | 你的任务 | "设计一个可扩展的状态管理方案" |
| A - Action | 技术方案 + 为什么 | "选择注册中心 + Hook 插件架构，而非硬编码 switch" |
| R - Result | 量化成果 | "新增类型开发量从 3 天降至 0.5 天" |
| X - X-Factor | 创新点/影响力 | "该方案被团队复用为其他服务的标准框架" |

### 3.2 量化表达标准

**必须量化的指标类型**：

| 类型 | 示例 | 如何获取 |
|------|------|----------|
| 性能提升 | "响应时间从 500ms 降至 80ms" | 从 APM/日志/压测报告获取 |
| 规模/体量 | "支撑 30+ 种类型 / 10+ 个服务接入" | 从代码中统计 |
| 效率提升 | "开发量从 3 天降至 0.5 天" | 根据经验估算 |
| 可靠性 | "投递成功率 99.9% / 状态一致性 100%" | 从监控获取 |
| 成本节省 | "减少锁相关 Bug 90%" | 根据经验估算 |

**量化的底线**：每条贡献至少有一个数字。如果实在无法量化，用对比描述：
- "从手动部署改为自动化" → "部署时间从 30 分钟缩短至 5 分钟"
- "优化了接口" → "接口响应时间降低 60%"

### 3.3 动词选择

**体现设计能力的动词**（3年经验应该用）：
> 设计 / 架构 / 主导 / 规划 / 抽象 / 封装 / 重构

**体现执行能力的动词**（辅助使用）：
> 负责 / 实现 / 开发 / 集成 / 对接 / 部署

**禁用的动词**（太弱/太虚）：
> ~~参与~~ / ~~协助~~ / ~~学习~~ / ~~了解~~（除非是次要职责的补充说明）

### 3.4 技术深度要求（3年经验标准）

| 层次 | 说明 | 是否达标 |
|------|------|----------|
| L1: 会用 | "用了 Redis 做缓存" | ❌ 1年水平 |
| L2: 会选型 | "选择 Redis 而非 Memcached，因为需要丰富数据结构" | ⚠️ 2年水平 |
| L3: 会设计 | "设计了三级缓存策略 + 缓存一致性方案" | ✅ 3年水平 |
| L4: 会抽象 | "抽象为通用缓存框架/Starter 供全公司使用" | 🌟 超出预期 |

**输出时确保每条核心贡献至少达到 L3 水平。**

### 3.5 反模式检查清单

输出最终简历前，逐条检查是否踩了以下坑：

| 反模式 | 问题 | 修正 |
|--------|------|------|
| 技术名词堆砌 | "用了 Spring Boot + Redis + MQ" 没有上下文 | 加上"为什么用"和"解决了什么" |
| 职责模糊 | "参与了系统开发" | 明确你做了哪个模块，用"我"而非"团队" |
| 没有结果 | 只说了做什么，没说效果 | 每条必须有量化结果 |
| 千篇一律 | 每条都是"负责 xxx 的开发" | 用不同角度：设计/优化/重构/封装 |
| 过度包装 | 吹嘘百万QPS但实际业务不到千级 | 如实描述，面试会被追问穿帮 |
| 只写CRUD | "负责用户模块增删改查" | 找技术难点：批量/并发/性能/一致性 |
| 缺少版本号 | "使用 Spring Boot" | 写 "Spring Boot 3.2"，体现技术时效性 |
| 行业术语过多 | 充满业务名词面试官看不懂 | 用通用技术语言描述，业务只做背景 |

### 3.6 ATS 关键词优化

简历会经过招聘平台的自动筛选系统（ATS），确保输出中包含目标岗位 JD 的核心关键词：

常见 Java 后端 ATS 关键词：
```
Java 17 / Spring Boot / Spring Cloud / 微服务 / MyBatis
Redis / 分布式锁 / 缓存 / MySQL / 索引优化
RabbitMQ / Kafka / 消息队列 / 异步
Docker / K8s / CI/CD / 容器化
设计模式 / 高并发 / 高可用 / 分布式事务
JVM / GC 调优 / 性能优化
RESTful / gRPC / API 设计
```
