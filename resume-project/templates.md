# 多版本输出与质量保障

## 第四步：多版本输出

根据不同使用场景，输出多个版本：

**版本 A：完整版（用于投递简历，PDF/Word）**
- 每条贡献 3-5 行
- 包含完整的 STAR-X 结构
- 总长度控制在 1-1.5 页内

**版本 B：精简版（用于 Boss 直聘/在线简历）**
- 每条贡献 1-2 行
- 只保留"做了什么 + 关键成果"
- 总长度控制在 300-500 字

**版本 C：口述版（用于面试自我介绍）**
- 每条贡献用一句话概括
- 突出最亮的 2-3 个点
- 为追问埋钩子（说一半留一半）

---

## 第五步：输出建议

根据项目分析结果，额外给出：
- 针对 3 年 Java 开发工程师的简历优化建议
- 建议补充或强化的技术点（参考目标行业/公司 JD）
- 项目中可以进一步挖掘的亮点
- 面试时可能被追问的方向 + 建议准备的答案框架
- 与目标岗位 JD 的匹配度分析（如果用户提供了 JD）

---

## 输出格式

最终输出的简历项目描述应为 Markdown 格式（方便预览），同时提供纯文本版本（方便复制到简历工具中）。语言默认中文，用户可要求英文。

**输出结构**：
1. 先输出完整版（版本 A）
2. 再输出精简版（版本 B）
3. 最后输出口述版（版本 C）+ 面试追问准备建议
4. 附上质量检查清单的通过结果

---

## 质量自检清单

输出前必须逐条对照以下清单，不通过则修改后再输出：

- [ ] 每条贡献都有量化数字（至少 1 个）
- [ ] 技术栈版本号准确（从 pom.xml 读取）
- [ ] 没有使用"参与""协助"等弱动词作为开头
- [ ] 每条贡献都包含"为什么"（技术选型理由）
- [ ] 没有行业黑话/只有本公司才懂的术语
- [ ] 总长度合理（完整版 1-1.5 页，精简版 300-500 字）
- [ ] 技术深度达到 L3+（会设计/会抽象，而非只是会用）
- [ ] 包含 ATS 关键词（Java 17/Spring Boot/Redis/微服务等）
- [ ] "我的定位"准确反映了实际角色（不夸大也不贬低）
- [ ] 面试时每条都能展开讲 3 分钟（有足够的细节支撑）

---

## 注意事项

- **只描述用户个人负责的部分**，不要把团队成果算在个人头上
- 不要编造不存在的功能或数据
- 如果项目较简单，如实描述，不要过度包装
- 量化数据如果无法从代码中获取，用 `[待补充: xx]` 标记，提醒用户自行填写
- 技术栈版本要准确，从配置文件中读取
- 如果项目有多个模块，可以拆分为多个子项目分别描述
- 如果 git 历史不可用，必须通过询问用户来确定个人职责边界
- 简历中的"负责"要名副其实——只写真正由你开发或深度参与的内容
- **子系统打开场景**：用户通常只打开一个微服务子系统目录（如 `xxx-product`），而非整个项目根目录。skill 必须能在这种场景下正常工作，不应要求用户打开完整项目。如需访问其他目录的代码，通过 Shell 工具的 `working_directory` 参数切换。
- **多系统场景**：不要把一个大项目写成一条臃肿的经历，按职责模块描述更清晰；但也不要拆太碎导致每条都浅，2-3 条为宜
- 子系统之间有技术关联（如共用网关、公共组件、统一 Starter）时，在项目简介中点明整体架构，再分条展开各子系统贡献
- **多目录模式**：每个目录是独立项目，各自产出一条项目经历；最终统一润色时注意体现技术成长线
- **基础设施贡献**：如果用户开发了 Starter / SDK / 公共组件，即使代码不在当前打开的目录中，也应该纳入简历，可以通过用户提供的路径去扫描
