# 扫描、阅读与出题工作流

## Step 1：扫描项目结构

```
Glob: **/pom.xml, **/*.java, **/*.lua, **/application*.yml
```

识别并记录：
- 模块划分（starter / core / api）
- 技术栈：Spring Boot 版本、Redis 客户端、AOP、SpEL 等
- 入口类、自动配置类（`*AutoConfiguration`）、SPI（`META-INF/spring/`）

输出简短「项目画像」（5-10 行），供用户确认分析范围。

## Step 2：深度阅读核心代码

按优先级阅读，**边读边记**（设计决策 / 亮点 / 风险）：

| 优先级 | 目标 | 关注什么 |
|--------|------|----------|
| P0 | 注解定义 + AOP 切面 | 拦截逻辑、SpEL 解析、异常转换 |
| P0 | 核心业务实现类 | 加锁/解锁流程、重入、续期 |
| P1 | Lua 脚本 | 原子性保证、参数含义、边界分支 |
| P1 | 配置类 + Properties | 默认值、条件装配 `@ConditionalOn*` |
| P2 | 工具类、SPI 接口 | 扩展点、策略模式 |
| P2 | 测试类 | 覆盖场景 = 边界情况线索 |

阅读时提取三类素材：
- **设计决策**：为什么用 X 不用 Y（需有代码或注释佐证）
- **技术亮点**：原子操作、防误删、可重入、Watchdog 等
- **潜在风险**：竞态、锁泄漏、SpEL 注入、异常吞没、配置缺失

## Step 3：阅读文档

```
Glob: README*, CHANGELOG*, docs/**, *.md
```

文档与代码冲突时，**以代码为准**，并在陷阱题中标注差异。

## Step 4：提取出题素材

整理「出题素材表」（内部使用，不必输出给用户）：

| 素材 | 来源示例 | 适合维度 |
|------|----------|----------|
| 选型理由 | Redisson vs SET NX、Lua vs 事务 | 设计决策 |
| 底层机制 | AOP 环绕通知、Lua EVAL 参数 | 原理深挖 |
| 防御逻辑 | try-finally 释放、leaseTime 兜底 | 边界情况 |
| 扩展接口 | LockStrategy SPI、自定义 Key 生成器 | 扩展方向 |
| 代码隐患 | 未校验返回值、线程中断未处理 | 陷阱题 |

## Step 5：按 5 维度生成面试题

每维度至少 2 题，全项目建议 **10-15 题**。维度定义：

| 维度 | 提问模式 | 出题要点 |
|------|----------|----------|
| **设计决策题** | 「为什么用 X 而不用 Y？」 | 从代码中找到实际选型及注释/实现依据 |
| **原理深挖题** | 「这个机制底层怎么实现的？」 | 追到 Redis 命令、Lua 脚本行级、AOP 切面调用链 |
| **边界情况题** | 「如果 XXX 场景会怎样？」 | 找防御逻辑、异常处理、超时/重入/宕机场景 |
| **扩展方向题** | 「要支持 XXX 你会怎么做？」 | 基于现有 SPI、接口、配置扩展点设计方案 |
| **陷阱题** | 「你考虑过 XXX 问题吗？」 | 指出潜在风险，并给出改进思路（需诚实标注是否为真实隐患） |

## Step 6：生成追问链

每题固定 **3 层递进**：

```
L1（确认理解）→ L2（原理/细节）→ L3（极端场景或方案对比）
```

示例路径：「分布式锁怎么实现的？」→「Lua 脚本为什么能保证原子性？」→「Redis 主从切换时锁丢失怎么办？」

## 出题质量检查清单

输出前逐项自检：

- [ ] 每题均有具体 `类名#方法名` 定位
- [ ] 参考答案含代码证据（非纯概念）
- [ ] 追问链 3 层且层层加深
- [ ] 5 个维度均有覆盖
- [ ] 陷阱题基于真实代码风险，非臆造
- [ ] 未把未读文件的内容当作事实
